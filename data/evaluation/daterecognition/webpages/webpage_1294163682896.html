The Component Interface - No... Its Not a Star Trek Device!

Derek Tomei | Jun 19, 2006 | Comments (16)
What is a Component Interface?

It's not something on the starship enterprise. First you have to understand what a component is. In its simplest description, it is a collection of PeopleSoft pages. Prior to version 8 of PeopleTools, a component was known as a Panel Group. A component is a page or set of pages that are similar in that they share related tables and keys. They are used for adding and updating data. They can also be used to view the data in an organized structure. More importantly, Components are responsible for enforcing the data entry rules and integrity. These rules are in the form of translate tables, prompt tables, or PeopleCode.

A Component Interface (CI) is an object within PeopleSoft's PeopleTools toolset. It is also a part of PeopleSoft's Integration technology (also known as Enterprise Integration Points or EIP's). The main purpose of the component interface is for interfacing with PeopleSoft applications from outside of the PeopleSoft environment or between different PeopleSoft modules so that the data will interface with the Component and maintain business logic and data integrity.

When you perform an insert statement directly into a table (outside of PeopleSoft), you bypass these rules. You still have some rules like unique key restraints; these types of rules are enforced by the database itself, but nothing on the level that PeopleTools offers.

So why is this important? Well you could have a field for which you want to only have specific values, like a Department ID or a JOBCODE. You might only want data that is contained on a specific list. You might only want data that is further refined, based upon other data contained within the other set of data that you are currently entering. Obviously this has a major impact when you are performing data conversion. Did that make your head spin?

You can read an App Engine / Component Interface guide on how to develop using the component interface within an application engine program.

The greatest Benefit of a Component Interface is that You don't have to re-code all of the work that has already been developed in People Tools for maintaining business logic and enforcing integrity rules. Why should you re-code functions such as converting PS_JOB rows, when there is already an incredible amount of rules set up in the table structure and PeopleTools? Why reinvent the wheel? Thus, the power of component interfaces.

Srinivasa Rama Naidu Kayala | Feb 9, 2009
Hi floresn,

There is definitely a relation between the app server(i'm not sure on what other servers it exerts pressure) and CI Load. I have done few bulk uploads into the Job Data Component. Its an yearly avtivity for us. Whenever we do it, we first inform our users about the down-time (we generally schedule it on non-working days) and shutdown the app server before doing upload. The actual volume is nearly 3000+ rows, but we do it in chunks of 100-500, to reduce the problems due to roll-back in case of unexpected errors.

Note:

1) More Load is experienced, If your upload data needs the Component to go into "Correct History" mode, which is the case with me. (example uploading back-dated data, which involves the Job Component to copy this data to further dated rows also)

2) Load depends on the complexity of your component: the number of tables involved and the amount of peoplecode written on it.

Also, is the CI being run online or through Application Designer? in my case it is the App Designer..

Rama Naidu.
